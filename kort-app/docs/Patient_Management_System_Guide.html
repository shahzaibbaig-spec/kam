1. Module Objective

The Patient Management System extends KORT into a hospital-ready clinical workflow where staff can register patients, admit them, assign doctor/ward/bed, record visits and diagnosis, issue prescriptions, and maintain a permanent clinical history.

Core objective in one line:
Search or register patient -> Admit if needed -> Doctor documents visit and diagnosis -> Create prescription -> Print/Download PDF -> Preserve full history and audit logs.

2. Scope and Integration

  • Built inside the existing KORT Laravel application; no change to Asset, Inventory, Procurement, and core UI architecture.
  • Uses existing authentication and role-permission framework (Spatie Permission).
  • Uses existing activity/audit logging conventions.
  • Uses existing blue/white modern Inertia UI flow.
  • Supports OPD, Emergency, and admitted follow-up workflows.

5. User Management and Password Authority

5.1 Who Can Add New Users?

New staff users can be added from Administration -> Users by roles having users.create. In the current matrix, this is primarily Super Admin and Hospital Admin.

5.2 Who Can Change User Access and Roles?

User profile updates require users.update. Role assignment changes are controlled by users.assign-roles and role permissions by roles.update.

5.3 Who Can Change Passwords?

6. Operating Procedures

6.1 Register New Patient (Reception / Admin)

Step 1: Open Patient Care -> Patient Search and search first using CNIC, patient number, name, or phone.
Step 2: If no existing record matches, go to Patients -> New Patient.
Step 3: Enter demographics, emergency contact, allergies, medical history, optional photo.
Step 4: Save; patient number is auto-generated when not manually entered.

CNIC is validated for Pakistani format (#####-#######-# or 13 digits) and stored in normalized format to prevent duplicates.

6.2 Fast Search Logic

  • Exact CNIC or exact patient number: opens patient profile directly.
  • Name/phone query with multiple matches: shows result list for selection.
  • Search is available on Patient List and dedicated Patient Search page.

6.3 Patient Admission Workflow

Step 1: Open patient profile.
Step 2: Click Admit Patient.
Step 3: Enter admission date/time, department, ward/room/bed, attending doctor, reason, initial condition.
Step 4: Save admission with status admitted.

Admission status lifecycle supports: admitted, discharged, transferred, cancelled.

6.4 Doctor Queue and Visit Handling

  • Doctors open Patient Care -> Doctor Queue to see admitted patients assigned to them.
  • From patient/admission context, doctor creates diagnosis entry which automatically creates linked visit record.
  • Visit captures complaint, vitals, notes, visit type, date/time, doctor, and optional admission link.

6.5 Diagnosis Workflow

Step 1: Open Create Diagnosis for patient.
Step 2: Fill visit details and chief complaint.
Step 3: Enter diagnosis, clinical notes, severity, and follow-up date.
Step 4: Save to permanently append diagnosis and visit history.

6.6 Prescription Workflow

Step 1: After diagnosis save, open Create Prescription.
Step 2: Add one or more medicine rows (name, dosage, frequency, duration, instructions).
Step 3: Add general instructions and printable notes.
Step 4: Save; prescription is linked to patient and visit with unique prescription number.

7. Prescription Print and PDF

Authorized users with patient-prescription.print can:

  • Open browser-print view (Print Prescription)
  • Download official PDF (Download PDF)

Prescription output includes:

  • Hospital/system name
  • Patient name, patient number, CNIC, age/gender
  • Date/time and doctor name
  • Diagnosis summary and follow-up date
  • Medicine table and instructions
  • Doctor signature area

8. Patient Profile and History Design

The profile page is a complete clinical dashboard per patient:

  • Demographics and contact information
  • Allergies and medical history
  • Current admission status
  • Past admissions list
  • Visit history
  • Diagnosis history
  • Prescription history with print/PDF access
  • Uploaded documents and notes

The History page renders a timeline-style longitudinal patient journey for clinical and compliance review.

9. Audit and Compliance Controls

All critical patient events are activity-logged, including:

  • Patient created / updated / archived
  • Admission created / updated / discharged
  • Visit created / updated
  • Diagnosis created / updated
  • Prescription created / printed / PDF downloaded
  • Patient document uploaded

This logging supports traceability, accountability, and post-incident review.

10. End-to-End Working Design (Operational Blueprint)

Front Desk Layer: Search-before-create, patient registration, admission initiation.
Clinical Layer: Doctor queue, visit capture, diagnosis, medication planning.
Treatment Output Layer: Prescription print and PDF generation for bedside and pharmacy use.
History Layer: Longitudinal timeline and profile retrieval for follow-up continuity.
Governance Layer: Role-restricted access + audit log monitoring.

13. Implementation Notes for Admin Team

  • Use role-based access only; avoid assigning broad rights to non-admin users.
  • Ensure CNIC entry standardization at front desk.
  • Use discharge summary consistently for inpatients.
  • Review audit logs routinely via Security section.
  • Keep doctor and ward assignment accurate to maintain queue quality.
